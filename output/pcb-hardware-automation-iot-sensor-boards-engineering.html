Key Takeaways

High-level engineering readout
PCB automation is no longer just about schematic capture: the pipeline now spans requirement capture, symbol generation, schematic synthesis, placement, and routing.
The strongest open-source pieces are the agent/tooling layers around KiCad, while the strongest product signals come from CELUS, Flux, and Altium on requirements and schematic assistance.
For IoT sensor boards, the hardest remaining problems are constraint propagation and layout closure, especially when mixed-signal, clearance, and layer-count constraints collide.

Open Source

4 repos / projects

Circuitron

open_source
GitHub · Agentic PCB design accelerator
Circuitron converts natural-language requirements into SKiDL scripts, KiCad schematics, and PCB layout files. The repo documents a multi-agent pipeline with part discovery, documentation lookup, code generation, validation, and ERC correction, all wired through MCP and containerized tooling.
Why it matters: this is the most complete open-source path I found for taking a textual board brief and producing editable KiCad artifacts instead of a text-only plan.

circuit-synth

open_source
GitHub · Python-defined circuits
circuit-synth represents circuits in code, then exports schematic and PCB-related outputs through the tool chain. That makes the design intent versionable and parameterizable, which is useful when you want to generate repeated sensor-board variants from one template.
Why it matters: code-first circuits are much easier to constrain, diff, and regenerate than free-form drawings, which is a good fit for IoT board families.

kicad-tools

open_source
GitHub · KiCad parsing and manipulation helpers
kicad-tools exposes machine-readable commands for parsing and modifying KiCad schematic and PCB files without launching the GUI. It is explicitly positioned as an agent-friendly layer for analysis, CI, and automated board edits.
Why it matters: any real PCB automation agent needs a structured way to inspect and rewrite existing design files, not just generate a first draft.

MCP KiCad Schematic API

open_source
GitHub · MCP server for schematic manipulation
This project wraps KiCad schematic operations behind MCP tools so AI agents can create components, wires, and hierarchical designs programmatically. It is not a full board synthesizer, but it supplies the plumbing that makes interactive design loops possible.
Why it matters: it is the clearest open-source bridge between an LLM agent and a real EDA file format.

Products

4 products / features

CELUS Design Platform

product
CELUS · AI-assisted electronics design platform
CELUS says its platform automates component search and schematic generation while capturing requirements and handing off BOM, footprints, and EDA data to downstream tools. The product framing is requirement-first: define blocks, evaluate solutions, and then transfer the result into the engineer’s EDA flow.
Why it matters: this is one of the strongest current product signals that requirements-to-schematic automation has moved past pure research.

CELUS Design Assistant

product
CELUS · Natural-language design assistant
The Design Assistant interprets text or sketches and turns them into structured functional diagrams. It is the clearest product-level example I found of users starting from a prompt such as a sensor-board brief and rapidly getting a system architecture.
Why it matters: it shows how natural language can seed a board design workflow before layout begins.

Flux Copilot

product
Flux · PCB design assistant
Flux Copilot is positioned as a schematic-first assistant. Flux’s own documentation says it still has limited understanding of PCB layout and trace positioning, which is a useful honesty check about where the product is today.
Why it matters: it is useful evidence that the product market is already comfortable with AI helping schematic work, but not yet fully automating board layout closure.

Altium Requirements Management

product
Altium · Requirements traceability for PCB design
Altium ties requirements, verification, and PCB design data together in a live workspace. The official documentation describes requirements import, AI-assisted requirement structuring, and traceability into the design itself.
Why it matters: requirements management is the control plane for requirement-to-PCB generation, even when the final placement and routing still need human oversight.

Why it matters: when a large research lab formalizes a PCB benchmark, it usually means the ecosystem is getting serious about machine-readable design workflows.

Why it matters: it shows commercial appetite for early-stage automation in electronics design, especially where teams want faster starts and cleaner handoff into EDA.
