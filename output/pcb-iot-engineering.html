Key Takeaways

High-level engineering readout
需求到“完整可投产 PCB 文件”的全自动链路已经出现原型，但当前最稳的落地方式仍然是“需求到原理图/BOM/约束，再由人审完成布局与可制造性收尾”。
可行实现路径已经比较清晰：自然语言需求 -> 组件/功能块检索 -> datasheet grounding -> 原理图合成 -> ERC/约束验证 -> KiCad/EDA 文件输出。真正的瓶颈在布局、SI/PI、封装与制造规则。
工程上最成熟的不是一个万能 agent，而是把 SKiDL/KiCad/约束检查器/零件数据库串成流水线。也就是说，IoT PCB 自动化更像“多工具编排”，而不是单模型端到端生成。

Open Source

3 repos / projects

Circuitron

open_source
Shaurya Sethi / GitHub · 2025
面向自然语言 PCB 设计的 agent 原型，能够把需求转成 SKiDL、原理图和 `.kicad_pcb` 等工件，已经非常接近“需求到 KiCad 文件”的自动化链路。
实现路径是“文本需求 -> SKiDL Python 电路描述 -> KiCad 工件生成 -> 迭代修正”。这说明完整 PCB 文件可以被自动生成，但质量仍依赖后续约束检查与人工审阅。

circuit-synth

open_source
circuit-synth / GitHub · 2026
一个以 Python 定义电路、再导出 KiCad 项目的开源栈，强调可编程电路描述和自动生成原理图/PCB。
它不是纯文本到 PCB 的黑盒，而是把设计抽象成代码，再接 KiCad 输出。这类“代码化电路描述 + EDA 导出”的模式很适合 IoT 传感板模板复用。

kicad-tools

open_source
rjwalters / GitHub · 2025
提供程序化解析和修改 KiCad schematic / PCB 文件的 Python 工具，适合作为 agent 的执行后端。
如果前端 agent 负责需求理解和约束生成，`kicad-tools` 这种库就能承担文件级编辑、规则修复和批量重写，是自动化 PCB 流水线里很关键的“文件操作层”。

Products

3 products / features

Flux Copilot

product
Flux · 2023-2026
目前最接近“AI 辅助硬件设计”的商业产品之一，主打在原理图阶段提供器件建议、连接解释和设计协助。
官方文档明确写到它当前主要理解 schematic，对 PCB layout 的理解仍有限。这和研究结论一致：真正难的是从原理图跨到布局与制造约束。

CELUS Design Assistant

product
CELUS · 2026
CELUS 已把自然语言需求输入接到了电子设计平台上，强调从需求理解、器件选择到 schematic/BOM 生成的加速。
它更像“需求到设计包”的前半段引擎：先拆功能块，再做元件匹配和系统设计，再交给 EDA 工具继续收尾。对 IoT 板来说，这是一条很现实的工程路径。

KiCad

product
KiCad Project · active
KiCad 本身不是 AI 产品，但它是当前开源自动化链路里最重要的目标格式和执行后端。
大量自动化项目最终都落在 KiCad 的 schematic 与 PCB 文件上，因为它开放、可脚本化、生态完整。现实里“自动生成 PCB 文件”通常意味着最终落到 KiCad-compatible 项目。
