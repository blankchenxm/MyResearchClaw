Gunar Schirner et al.
ASP-DAC 2008 code-generation mpsoc synthesis
摘要Summary
这篇 ASP-DAC 论文是较早但仍然很有参考价值的例子：从抽象系统规格自动生成依赖硬件的软件。它给出的核心工程启发是，一旦硬件契约和通信结构被放进形式化中间表示，代码生成就会变得可控。这个结论与今天跨 ESP32、STM32、nRF 的 MCU 自动适配问题高度一致，因为真正困难的不是输出 C 代码，而是如何把引脚、总线和外设假设稳定地带过不同芯片族。它支持这样一种路线：自动固件生成应该建立在结构化设备模型之上，而不是单纯依赖 prompt 驱动的代码补全。
This ASP-DAC paper is an early but still relevant example of generating hardware-dependent software from abstract specifications. The important engineering lesson is that code generation becomes tractable once the hardware contract and communication structure are expressed in a formal intermediate representation. That maps well to today’s cross-chip MCU problem, where the hard part is not emitting C code, but carrying pin, bus, and peripheral assumptions across families. It supports the idea that automatic firmware generation should be built around structured device models rather than purely prompt-driven code completion.

Jinsik Kim et al.
ISLPED 2009 ota flash-update embedded
摘要Summary
这篇论文提出面向 Flash 型嵌入式系统的渐进式远程固件更新方法，目的是降低受限设备整包替换的代价。它今天依然相关，因为很多“自动化烧录”方案只关注代码生成，却忽略了带宽、能耗和 Flash 磨损这些现实部署成本。论文把更新问题建模成增量式系统问题，而不是一次性整体重刷，这更接近 IoT 节点的实际运行条件。对于电池供电的小设备来说，这种思路尤其重要。
This paper presents progressive firmware update for flash-based embedded systems, reducing the burden of replacing whole images on constrained devices. Its relevance today is that many cross-MCU automation claims ignore the cost of bandwidth, energy, and flash wear when deploying generated images. By treating update as an incremental systems problem, the paper gives a more realistic model for field deployment. That is particularly useful for battery-powered IoT nodes where full reflashing is often the wrong operational primitive.

This paper targets extremely constrained wireless sensor nodes and proposes OTA updating without reboot and without extra flash reservation. The main lesson is that update automation is dominated by memory layout, availability, and runtime continuity, not only by compiler support. In other words, the last mile of automated flashing is mostly systems engineering.

This paper studies how to compile platform-independent timed models into platform-specific embedded code while preserving timing semantics. Its significance for modern MCU automation is that it formalizes the boundary between application intent and board-specific implementation, which is exactly the boundary a firmware agent has to cross when targeting ESP32, STM32, or nRF devices. Rather than treating firmware generation as free-form coding, the paper treats it as constrained synthesis against timing models and deployment assumptions. That perspective remains useful for any requirement-to-firmware pipeline that needs deterministic behavior instead of approximate code generation.

It is a reminder that automation without secure update infrastructure is incomplete.

Christian Niesler et al.
NDSS 2021 hotpatching embedded-security real-time
摘要Summary
HERA 同样研究嵌入式实时软件的在线更新，但更强调在严格的时序和内存约束下如何保持系统正确性。对 MCU 自动化来说，它的重要性在于：如果一条端到端自动流水线连在线设备的稳定更新都做不到，那它就谈不上真正可部署。论文说明，小型控制器上的自动化部署绝不只是编译问题，而是涉及内存布局、补丁兼容性和执行安全的一整套系统问题。这一点直接关系到“跨多类 MCU 自动烧录”是否真的可信。
HERA also studies live updating for embedded real-time software, but from the angle of preserving correctness under stringent timing and memory constraints. For MCU automation, this paper matters because an end-to-end pipeline is only believable if it can update running devices without destabilizing them. The work makes clear that automated deployment on small controllers is not simply a compiler problem; it is a systems problem involving memory layouts, patch compatibility, and execution safety. That is directly relevant to any claim of autonomous flashing across multiple MCU families.

NDSS 2021 rehosting microcontroller analysis
摘要Summary
这篇论文展示了如何通过移除或适配硬件依赖，让 MCU 固件在通用硬件上运行。它对自动化固件生成特别重要，因为一旦能在离板环境里先运行和调试生成结果，就能在真正接触 ESP32、STM32、nRF 硬件之前完成大量验证。这样会显著缩短自动生成代码的迭代闭环，也让 agent 式固件生成更接近可用工程系统。现实里，rehosting 是把固件自动化从玩具 demo 推向可扩展实践的重要支撑能力。
This paper shows how MCU firmware can be rehosted on commodity hardware by neutralizing hardware dependencies. The immediate relevance is to automated firmware generation loops: if you can run generated firmware off-target, you can validate and debug it before touching a real ESP32, STM32, or nRF board. That shortens the iteration cycle and makes autonomous code generation much more realistic. In practice, rehosting is one of the strongest enablers for scaling firmware agents beyond toy examples.

摘要Summary
RapidPatch 关注的是量产环境里真正棘手的问题：如何在不重启设备、也不破坏实时性的前提下更新嵌入式固件。它之所以与本题相关，是因为“自动生成固件”只是前半段，后半段还包括如何把生成结果安全地下发、修补并维持在设备上。论文把讨论重点从一次性刷写推进到更细粒度的生命周期管理，这比单纯烧录更接近工业 MCU 设备的真实运维方式。对跨平台自动化来说，它提示我们：部署约束的重要性并不亚于代码综合本身。
RapidPatch tackles a production-facing firmware problem: how to update real-time embedded code without rebooting the device or violating timing guarantees. That matters here because automated firmware generation is only half the story; the generated image must also be safely delivered, patched, and maintained on deployed devices. The paper moves the discussion from one-shot flashing toward fine-grained lifecycle management, which is much closer to how industrial MCU fleets operate. For cross-platform automation, it highlights that deployment constraints can be as important as code synthesis itself.

AutoPatch 更进一步，它尝试自动为实时嵌入式设备生成热补丁，而不是每次都依赖工程师手工修复。它的重要性在于把自动化从“初始代码生成”推进到了“部署后维护”阶段。对于异构 MCU 设备群来说，难点不仅在于生成首版固件，还在于如何生成既安全又不破坏时序和状态连续性的增量补丁。它是一个很明显的信号：固件自动化正在从编译期辅助走向运行期生命周期管理。

For heterogeneous MCU fleets, the difficult part is not just generating an initial firmware image, but generating safe deltas that respect timing and state continuity. This paper is one of the clearest signals that the firmware automation stack is moving from compile-time assistance toward runtime lifecycle automation.

Huanqi Yang et al.
SenSys 2026 llm firmware-generation iot
摘要Summary
AutoEmbed 与你关心的问题几乎是正面对齐的：能不能把通用嵌入式 IoT 需求自动变成可工作的固件。公开页面给出的系统结构是四阶段流水线，依次包含 library solving、knowledge generation、prompt assembly 和 LLM-driven code synthesis，也就是说它并不是让模型对着空 prompt 直接裸写代码。这个架构很关键，因为在 ESP32、STM32、nRF 这类平台上，真正困难的部分往往是先选对硬件相关库、再把足够扎实的上下文喂给生成器，而不是最后几步代码输出本身。作为 SenSys 2026 的论文，它是目前最直接的证据之一，说明“需求到固件”的自动化已经从零散 demo 进入到较严肃的系统研究阶段。
AutoEmbed is directly aligned with the question of whether a generic embedded IoT requirement can be turned into working firmware with LLMs. The system is described as a four-stage pipeline covering library solving, knowledge generation, prompt assembly, and LLM-driven code synthesis, which means it is not just asking a model to emit code from a blank prompt. That architecture matters because the hard part in ESP32, STM32, and nRF automation is selecting the right hardware-specific libraries and assembling enough grounded context before generation starts. As a SenSys 2026 paper, it is one of the clearest recent signals that requirement-to-firmware automation for embedded IoT has moved from scattered demos into serious systems research.
