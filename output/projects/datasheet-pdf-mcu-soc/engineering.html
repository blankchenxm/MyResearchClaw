这条路线已经形成比较清晰的工程拆分：图表/曲线抽取、时序图理解、表格与技术规格抽取、以及最终的 datasheet-to-code / traceability 映射。Ring 1 有 ChartOCR 这类 paper-linked 代码；Ring 2 开始出现面向 spec sheet 的产品化抽取；但对于 RK3588 或 STM32 这种超长 datasheet，跨页表格、图文对齐和 vendor-specific 图例仍然是主要难点。
This route has already decomposed into recognizable engineering stages: chart/curve extraction, timing-diagram understanding, table/spec extraction, and final datasheet-to-code or traceability mapping. Ring 1 includes paper-linked code such as ChartOCR; Ring 2 shows spec-sheet-oriented extraction products; but for long vendor datasheets like RK3588 or STM32, cross-page tables, figure-text alignment, and vendor-specific diagram conventions remain the main bottlenecks.
最可行的落地方案不是单模型端到端读懂整份 datasheet，而是多阶段 pipeline：版面分解 -> OCR/表格 -> 图表/时序专用解析 -> 规格 schema 化 -> 检索与问答。
The most viable deployment path is not one giant model that reads the whole datasheet end to end, but a multi-stage pipeline: layout decomposition -> OCR/tables -> specialized chart/timing parsing -> schema normalization -> retrieval and QA.
当前的主要缺口是：芯片物理结构图、性能曲线和时序图通常没有统一标注标准；同一 datasheet 里还会混有跨页表、脚注条件和图中嵌字，这些都让通用 PDF parser 很难稳定工作。
The main gaps are that physical structure diagrams, performance curves, and timing diagrams rarely follow one annotation standard; the same datasheet also mixes cross-page tables, footnote conditions, and embedded figure text, all of which make generic PDF parsers unstable.

ChartOCR / DeepRule

paper-linked
Chart data extraction code
ChartOCR 提供了从图表图像恢复结构化数值表的公开代码，是 datasheet 曲线图抽取最直接可复用的 paper-linked 资产之一。
ChartOCR provides public code for recovering structured numeric tables from chart images, making it one of the most directly reusable paper-linked assets for datasheet curve extraction.
它的实现逻辑是把 OCR、关键点检测和图表类型特定解码组合起来，适合拿来做 datasheet 性能曲线的专用解析器。
Its implementation combines OCR, keypoint detection, and chart-type-specific decoding, making it a good building block for a dedicated datasheet-curve parser.

Independent engineering solutions Independent engineering solutions

CHARTextract

tool
Rule-based chart extraction
CHARTextract 代表了一条更偏工程的路线：用可调规则快速逼近图表抽取需求，而不是完全依赖大模型。
CHARTextract represents a more engineering-oriented route: quickly refining chart extraction behavior with adjustable rules instead of relying entirely on a large model.
对于 datasheet 场景，这种方法的价值在于你可以针对某家芯片厂商的固定图例和版式做定制，而不是追求全领域通用。
In datasheet settings, this matters because you can customize for a vendor’s recurring legends and layouts instead of chasing universal generalization.

Specmake spec-sheet extraction

product
Spec-sheet to structured output
Specmake 直接把 spec sheet 解析定位到 PDF -> JSON / Excel 的产品问题，这和芯片 datasheet 抽取的落地方向非常接近。
Specmake frames spec-sheet parsing directly as a PDF-to-JSON/Excel product problem, which is very close to the practical direction of chip-datasheet extraction.
它说明真正工程化时，用户要的往往不是“理解论文意义”，而是稳定拿到结构化规格表和可追溯来源。
It highlights that in real deployments, users often want stable structured specs with source traceability rather than abstract document understanding.

AI-on-Demand technical spec extraction

prototype
Engineering PDF extraction prototypes
这个项目把技术规格 PDF 的信息抽取拆成多种方法组合，说明工业文档抽取通常不是单一路径。
This project breaks technical-spec PDF extraction into multiple methods, showing that industrial document extraction is usually not a single-method problem.
对于芯片 datasheet，这种组合式思路尤其合理，因为文本、表格、图表和 timing diagram 本来就需要不同模型和后处理。
For chip datasheets, this combinational approach is especially appropriate because text, tables, charts, and timing diagrams naturally require different models and post-processing.
